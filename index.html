WebAuthn, multi-chat, slash commands, streaming.

Reemplazo de Telegram para operar Hermes en producción. Auth con passkeys de salida, múltiples conversaciones simultáneas con SSE streaming token-a-token, autocomplete de comandos, y compatibilidad total con la API server nativa del gateway.

— Overview

Por qué construir esto

Telegram resuelve push notifications móviles bien. No resuelve UX para sesiones paralelas, historial navegable, ni interfaces especializadas para skills, cron y aprobaciones.

⌘

Multi-chat real

Cada sesión es una pestaña independiente con su propio stream EventSource. Cambiar de tab no corta los streams: siguen corriendo en background y avisan cuando terminan.

/

Slash commands

Pulsar / abre un palette con todos los comandos del registry de Hermes. Sourced en vivo desde /v1/capabilities con fallback local sincronizado.

⊕

Auth de salida

WebAuthn con passkeys más recovery codes. Sin password que robar. Cookie HttpOnly con SameSite=Strict, CSRF double-submit, rate limiting y audit log.

⟶

Compatibilidad total

Cero modificaciones al backend. La webapp es 100% cliente del API server existente. Si actualizas Hermes, sigue funcionando sin tocar código.

▢

Dark Enterprise

Mismo lenguaje visual que Vintek: minimalista, denso en información, tipografía mono para datos técnicos. Optimizado para sesiones largas sin fatiga visual.

⌨

Keyboard-first

⌘K switcher de sesiones, ⌘N nueva, ⌘↵ enviar, Esc cancelar stream activo.

Phase 1
— Seguridad

Hardening de salida

La app vive en el VPS detrás de TLS y autentica desde el primer despliegue. La superficie expuesta a Internet es solo nginx. El gateway nunca sale de loopback.

— principio

Defensa en profundidad. Confianza explícita en cada frontera.

Cada capa valida lo que recibe de la anterior. La API key del gateway nunca llega al cliente. Las queries a SQLite siempre van scoped por user_id. Los logs se redactan antes de escribir.

auth
WebAuthn (passkeys) + recovery codes hasheados con Argon2id. UV required, counter validado.
sesión
Cookie HttpOnly · Secure · SameSite=Strict · 24h idle · 7d max · revocación inmediata.
csrf
Double-submit cookie con verificación en cada mutación.
db
SQLite con SQLCipher (clave separada de la app). Permisos 0600. Wrapper de scoping con tests.
transporte
TLS 1.3 only · HSTS preload · CSP estricto · COOP/COEP · headers completos.
rate limit
Login 5/15min/IP, recovery 3/h, lockout escalonado, alertas a Telegram.
audit
Tabla append-only. Eventos: login, logout, recovery, credential add/remove. Retención indefinida.
supply chain
pnpm con lockfile, post-install whitelist, audit en CI, deps mínimas y auditadas.
secrets
.env con permisos 0600, validación Zod al boot, redacción en logs vía Pino.
— Arquitectura

Cómo se conecta al gateway

La webapp habla al API server vía loopback. Sin proxy intermedio, sin estado server-side propio del chat, sin riesgo de drift de versiones.
